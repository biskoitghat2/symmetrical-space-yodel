## ✅ باگ #11: دوبار حساب شدن چک در فاکتور

### 📍 مسیر
`store/dataStore.ts` - متدهای `addCheck` و `addInvoice`

### 🔴 مشکل
**سناریو:**
1. کاربر چک دریافتی ثبت می‌کند (منو چک‌ها → ثبت چک)
2. `addCheck` اجرا می‌شود و یک `CustomerTransaction` ایجاد می‌کند
3. مانده مشتری آپدیت می‌شود (مثلاً -1,000,000 ریال = بستانکار)
4. کاربر فاکتور فروش ثبت می‌کند و `paidCheckAmount = 1,000,000` وارد می‌کند
5. `addInvoice` اجرا می‌شود و **دوباره** یک `CustomerTransaction` برای همان چک ایجاد می‌کند!
6. مانده مشتری دوباره آپدیت می‌شود (مثلاً -2,000,000 ریال)

**نتیجه:** مشتری دو برابر بستانکار می‌شود! 💥

### 🟢 راه‌حل
```typescript
// ❌ BEFORE (Wrong):
if (invoice.paidCheckAmount > 0) {
  // Create customer transaction for checks
  const checkTrx: CustomerTransaction = {
    // ... creates duplicate transaction
  };
  await DatabaseService.addCustomerTransaction(checkTrx);
}

// ✅ AFTER (Fixed):
// Checks are already saved with their own customer transaction via addCheck()
// We should NOT create another transaction here
// The paidCheckAmount in invoice is just for record-keeping
```

**کد حذف شده:**
- حذف کامل بخش `if (invoice.paidCheckAmount > 0)`
- حذف کامل بخش `if (checkData && checkData.number)` (backward compatibility)

### 📊 تاثیر
- ✅ مانده مشتری حالا صحیح است
- ✅ چک فقط یکبار در کاردکس نمایش داده می‌شود
- ✅ محاسبات مالی دقیق است

### ⚠️ نکته مهم
**جریان صحیح:**
1. ابتدا چک را در منو "چک‌ها" ثبت کنید
2. سپس در فاکتور، فقط مبلغ `paidCheckAmount` را وارد کنید (برای نمایش)
3. سیستم دیگر تراکنش جدید ایجاد نمی‌کند

**محدودیت:**
- فعلاً ارتباط مستقیم بین فاکتور و چک وجود ندارد
- `paidCheckAmount` فقط یک عدد است، نه لینک به چک خاص
- این موضوع در باگ #13 بررسی شده است

## ✅ باگ #12: حذف تراکنش به جای ثبت برگشت

### 📍 مسیر
`store/dataStore.ts` - متد `updateCheckStatus` (بخش RETURNED)

### 🔴 مشکل
**سناریو:**
1. چک دریافتی ثبت می‌شود → تراکنش A ایجاد می‌شود
2. چک را PASSED می‌کنید → تراکنش A آپدیت می‌شود
3. چک برگشت می‌خورد → تراکنش A **حذف** می‌شود! ❌

**مشکل حسابداری:**
- طبق اصول حسابداری، تراکنش‌ها نباید حذف شوند
- باید یک تراکنش برگشتی (Reversal) ثبت شود
- کاردکس باید تاریخچه کامل را نشان دهد:
  - دریافت چک: -1,000,000
  - برگشت چک: +1,000,000
  - مانده نهایی: 0

**کد قبلی:**
```typescript
// ❌ WRONG: Deletes transaction
await DatabaseService.deleteCustomerTransaction(existingTrx.id);
```

### 🟢 راه‌حل
```typescript
// ✅ FIXED: Create a reversal transaction instead
const reversalTrx: CustomerTransaction = {
  id: crypto.randomUUID(),
  customerId: check.customerId,
  date: new Date().toLocaleDateString('fa-IR-u-nu-latn'),
  time: getCurrentTime(),
  type: 'PAYMENT_CHECK',
  description: `برگشت چک ${check.number} - ${check.bank}`,
  amount: existingTrx.amount,
  isDebtor: !existingTrx.isDebtor, // Opposite of original
  refId: check.id,
  refType: 'CHECK'
};

await DatabaseService.addCustomerTransaction(reversalTrx);
```

### 📊 تاثیر
- ✅ تاریخچه کامل در کاردکس نمایش داده می‌شود
- ✅ Audit trail حفظ می‌شود
- ✅ مطابق با اصول حسابداری است
- ✅ قابل ردیابی و رسیدگی است

## ⚠️ باگ #13: قطع ارتباط بین فاکتور و چک (نیاز به بازطراحی)

### 📍 مسیر
معماری کلی سیستم - ارتباط بین `Invoice` و `Check`

### 🔴 مشکل
**معماری فعلی:**
```typescript
interface Invoice {
  // ...
  paidCheckAmount: number;  // ❌ فقط یک عدد است!
  // هیچ لینکی به چک(های) واقعی وجود ندارد
}
```

**مشکلات:**
1. **حذف فاکتور:** اگر فاکتوری که با چک تسویه شده حذف شود، چک همچنان در سیستم باقی می‌ماند
2. **برگشت چک:** اگر چکی برگشت بخورد، فاکتور مرتبط اطلاع پیدا نمی‌کند
3. **گزارش‌گیری:** نمی‌توان فهمید کدام چک برای کدام فاکتور استفاده شده
4. **داده‌های یتیم:** چک‌ها و فاکتورها به صورت مستقل هستند

### 🟡 راه‌حل پیشنهادی (نیاز به تغییر معماری)

#### گزینه 1: اضافه کردن آرایه checkIds به Invoice
```typescript
interface Invoice {
  // ...
  paidCheckAmount: number;
  checkIds?: string[];  // ✅ لینک به چک‌های واقعی
}
```

**مزایا:**
- ساده و سریع
- Backward compatible

**معایب:**
- نیاز به آپدیت چک‌ها هنگام حذف فاکتور
- نیاز به آپدیت فاکتور هنگام تغییر وضعیت چک

#### گزینه 2: جدول واسط InvoiceCheckLink
```typescript
interface InvoiceCheckLink {
  id: string;
  invoiceId: string;
  checkId: string;
  amount: number;
  createdAt: string;
}
```

**مزایا:**
- معماری صحیح و قابل گسترش
- یک فاکتور می‌تواند چند چک داشته باشد
- یک چک می‌تواند برای چند فاکتور استفاده شود (split)

**معایب:**
- نیاز به تغییرات گسترده
- Migration پیچیده

#### گزینه 3: استفاده از CustomerTransaction به عنوان لینک
```typescript
// از refId و refType استفاده کنیم
// هر تراکنش می‌تواند به هم فاکتور و هم چک لینک شود
```

**مزایا:**
- از ساختار موجود استفاده می‌کند
- نیاز به تغییر کم

**معایب:**
- پیچیده برای query
- ممکن است گیج‌کننده باشد
